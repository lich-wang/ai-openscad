# AI OpenSCAD 产品文档

[English](PRODUCT.md)

## 产品定位

AI OpenSCAD 是一个 Agent 风格的 CAD 工作台，面向希望获得可打印或可检查 3D
模型、但更愿意先用自然语言描述结果的用户。产品会将用户需求转换成
OpenSCAD 代码，在浏览器中渲染草稿模型，通过视觉评审发现建模问题，并帮助
用户持续迭代，直到模型可以导出。

核心承诺是：

> 文本到 OpenSCAD，OpenSCAD 到模型，模型到视觉评审，评审到更好的下一版草稿。

## 目标用户

- 能描述所需物体，但不想手写全部 CSG 代码的 maker。
- 想快速得到第一版草稿，同时保留代码编辑出口的 OpenSCAD 用户。
- 需要快速生成 STL 进行检查的产品或硬件原型设计者。
- 中文和英文用户；应用跟随浏览器 locale，并在模型反馈和注释中沿用用户语言。

## 当前产品界面

### 工作台布局

首屏是可直接工作的应用，而不是落地页。整体分为三列：

- 左侧控制面板：基础设置、有上限的自动迭代控制和项目导入导出在前，新建模型操作和本地模型列表在后。
- 中间 Agent 面板：流水线箭头阶段、类似 Codex 桌面版的对话式运行流、需求输入、
  工作流操作和高级 OpenSCAD 代码编辑。
- 右侧结果面板：大型可交互 STL 预览、固定多角度评审视图和资产下载。

### 主流程

1. 用户写下需求，例如六格收纳盒或 30 ml 杯子。
2. 用户点击 **生成**。
3. 代码模型将 OpenSCAD 实时流式写入中间对话运行流。
4. 完整代码输出后，运行流默认折叠代码预览，渲染适配器在浏览器中编译生成的
   OpenSCAD。编译失败，或虽然产出 STL 但 OpenSCAD 诊断显示几何无效/缺失的危险渲染诊断，
   都会使用失败代码和可读诊断直接触发有上限的编译修复文本生成。
5. 完成干净渲染，或编译修复成功后得到干净渲染时，应用在右侧结果面板显示 STL
   大型可交互预览，用户可以直接旋转、缩放和平移模型。
6. 应用从 STL 捕获十四张 PNG：六个正交方向（正视图、后视图、左视图、右视图、俯视图、
   仰视图）和围绕模型的八个等轴测方向。
7. 如果未启用有上限的自动迭代，用户在干净渲染后点击 **评审**。如果用户在点击 **生成**
   前预设了自动迭代次数，应用会在该次有边界运行中，在每次干净渲染后自动执行视觉评审。
8. 视觉模型检查多角度视图是否满足原始需求，并返回总结、问题列表、作为评审结论的置信度和修正提示词。
9. 评审记录会把置信度百分比和总结一起展示。该次评审的修正提示词只显示一次，然后进入输入框，
   供手动迭代时编辑；提示词需要具体说明应该修改哪些 OpenSCAD 区域或几何关系。
   评审置信度只作为评审结论展示，不复制进修正提示词，也不发送给文本 LLM。
10. 用户点击 **再次迭代**，将最新已接受/已渲染的 OpenSCAD、原始需求、
   评审发现和可编辑迭代意见一起发送给 LLM，生成并渲染下一版草稿。如果用户在点击 **再次迭代**
   前预设了自动迭代次数，应用可以继续执行有上限的后续评审和修订循环，直到达到目标置信度或耗尽迭代次数。
11. 用户满意后点击 **最终导出**。
12. 应用将模型标准化为最终精度，并下载 `.scad` 和 `.stl` 文件。

### 用户控制规则

- 视觉评审默认只做评审，不能自动调用文本 LLM，也不能启动基于图片的自动重写循环；
  唯一例外是用户明确点击 **生成** 或 **再次迭代**，且自动迭代次数大于零，从而启动有上限的置信度运行。
- 在有上限的置信度运行中，每次干净渲染后都会进入视觉评审。如果评审置信度达到或超过用户预设目标置信度，
  本次运行以该评审结论停止。如果置信度低于目标且仍有自动迭代次数，应用可以用最新已渲染 OpenSCAD、
  原始需求、评审总结、问题列表和修正提示词发送一次后续文本修订，然后再次渲染和评审。
  达到目标置信度、耗尽自动迭代次数，或遇到 provider/编译/评审错误时，本次运行停止。
- 每次后续自动修订前，应用都会保留当前已渲染且已评审草稿的 checkpoint。如果新渲染草稿的评审置信度
  低于该 checkpoint，应用必须先恢复 checkpoint 的代码、STL、十四张视图、渲染证据和评审状态，
  再继续任何修订。恢复后应用会对该 checkpoint 重新执行一次视觉评审，获取当前修正提示词；
  后续自动修订必须基于恢复后的模型和新评审意见，而不能基于置信度降低的草稿。失败的后续修订仍然
  计入自动迭代次数；回退和重新评审不会返还该次数。
- 评审置信度是展示和保存的结论，不是提示词输入。它绝不能被插入可编辑修正提示词，也不能包含在生成或修订
  文本 LLM 提示词中。文本提示词可以包含评审总结、问题列表和修正提示词，但不能包含数值置信度或置信度阈值指令。
- 用户可以预设目标置信度百分比和有上限的自动迭代次数。默认自动迭代次数为零，以保持当前手动评审和迭代流程不变。
  次数含义是用户动作开始后的自动后续文本修订次数：**生成** 的初始草稿不计数，用户点击 **再次迭代**
  触发的那次修订不计数；每次评审后的自动后续修订计一次。
- 目标置信度控件是百分比滑杆，英文界面可访问名称为 `Target confidence`，中文界面使用本地化等价名称，
  范围 1-100，步进 1，默认 85。自动迭代控件是整数步进器/输入框，英文界面可访问名称为
  `Auto iterations`，中文界面使用本地化等价名称，范围 0-5，步进 1，默认 0。
  已持久化但超出范围的值在使用前会被夹取到这些范围。存储的目标值是百分比，并以
  `confidence >= target / 100` 与标准化评审置信度比较。
- 可交互预览只用于人工检查。拖动、缩放或平移预览不能改变十四张固定评审图片，
  也不能影响发送给视觉 provider 的 payload。
- 浏览器编译失败和危险渲染诊断在有代码模型 key 时必须直接触发有上限的 text-to-OpenSCAD 编译修复循环。
  危险诊断包括忽略未知变量或模块、未定义运算、数字或向量参数转换失败、include/use 文件缺失、
  NaN 或非有限几何，以及其他意味着捕获到的 STL 不完整或不符合源码意图的 OpenSCAD warning/error。
  每个编译步骤最多允许两次自动编译修复文本请求；达到上限后停在诊断状态。
  如果缺少代码模型 key，应用以可读诊断停止，并保留可编辑的手动重试指导。
- 编译修复循环适用于当前正在执行的编译步骤，无论它来自首次生成、手动重新渲染、接受修订、
  用户确认的再次迭代，还是有上限置信度运行中的自动后续修订。两次编译修复计数按每个编译步骤单独计算，
  并且独立于评审驱动的自动迭代次数。
- 这只借鉴经过验证的 text-to-OpenSCAD 工作流中有价值的部分：编译证据可以自动修复无效代码，
  渲染图片证据默认仍只作为用户控制的评审上下文，除非用户选择带可见目标和次数上限的置信度运行。
- 用户可以在再次生成前编辑修正提示词。
- 待确认修订必须先接受并渲染，才能导出项目。
- 接受修订会清除旧评审，强制新渲染模型重新评审。
- OpenSCAD 代码始终可以通过高级代码面板编辑。
- Agent 面板顶部以循环流水线箭头展示当前工作流阶段：代码生成 -> 模型渲染 ->
  模型评审 -> 下一轮迭代。
- 视觉评审输出显示在中间 Agent 运行流里，修正提示词会复制到可编辑输入框；
  右侧结果面板不重复展示评审文本、编译日志或历史。
- 中间运行流使用类似 Codex 桌面版的对话记录，在每次 LLM 交互前后展示记录。
  用户指令、流式 AI 输出、渲染工具进度、渲染完成、评审输出和迭代事件都作为
  独立记录展示。
- 需求输入区的主按钮随项目状态变化：渲染前是生成，渲染后是评审，评审后是再次迭代。
- 左侧设置区允许用户在点击 **生成** 或 **再次迭代** 前设置目标评审置信度和自动迭代次数。
  这些控制只影响未来由用户启动的运行；空闲时修改它们不会改变已有评审结论或提示词记录。
- token 估算和重复的就绪状态不在常规工作台界面显示，为模型列表和多角度视图留出更多空间。
- AI 提示词记录不属于常规工作台界面。

工作台验收标准：

- 桌面端左侧和窄屏堆叠布局中的顺序都保持稳定：包含目标置信度和自动迭代控件的基础设置、
  项目导入导出、新建模型按钮、本地模型导航列表。本地模型列表过长时在列表内部滚动，并且它不同于运行历史。
- “没有 Key？”邀请提示中的邀请图片按源图 50% 尺寸展示，保持源图比例，并受可用视口宽度约束；
  不应被裁切成正方形，也不应被放大到导致模糊。
- token 估算从常规工作台界面移除。处理中进度、待确认修订警告和错误仍通过 Agent
  阶段条与中间 Agent 运行流展示。
- 阶段条只用于展示信息，不可点击。它固定包含三个箭头阶段：代码生成、模型渲染和模型评审。
  每个阶段可以展示等待、处理中、已完成或阻塞/错误状态；生成后自动渲染期间，模型渲染阶段应显示为处理中。
  视觉样式应暗示评审反馈可以回到代码生成，进入下一轮迭代。
- 开始再次迭代时，当前轮次的阶段状态会重置：代码生成重新进入处理中，模型渲染等到代码完成后开始，
  模型评审等到新的渲染视图完成后再开始。
- 输入区主按钮在生成、渲染、自动置信度运行、自动编译修复、评审或导出期间禁用。没有当前干净十四视图渲染时显示“生成”，
  只有十四个当前渲染视图 key 全部非空、渲染诊断干净、当前草稿没有进行中或失败的修复，且没有当前评审时显示“评审”。
  只有当前干净十四视图渲染已评审且没有待接受修订时显示“再次迭代”。
  缺少输入、provider key 失败、编译失败和评审失败都显示在中间运行流，不改变右侧结果面板的职责。
  存在待确认修订时，输入区显示接受提示，不显示“生成”“评审”或“再次迭代”主按钮；
  用户必须先接受或拒绝修订，才能继续主流程。
- 有上限的置信度运行进行中时，生成、评审、再次迭代、重新渲染、最终导出、接受修订、拒绝修订、
  项目导入、项目导出、新建模型、本地模型导航，以及目标置信度和自动迭代控件都应为禁用或只读状态。
  这样可以避免过期的异步响应修改另一个项目或启动隐藏 provider 调用。本次运行停止后，这些控件根据最新项目状态恢复正常可用性。
- 如果生成或手动编辑代码编译失败，或虽然编译完成但诊断显示局部/无效几何，失败草稿不能进入视觉评审或导出。
  当前轮次中的旧评审、旧 STL 和旧视图图片都不能继续用于该失败草稿的评审/导出。
  中间运行流显示可读诊断，并在有代码模型 key 时立即开始有上限的编译修复生成。
  若修复次数耗尽或缺少 key，输入区保留可编辑诊断指导，供用户手动重试，同时保留当前失败的
  OpenSCAD 代码以便检查/编辑。
- 自动编译修复期间，代码生成阶段显示为处理中，模型渲染阶段等到修复代码流式完成后再开始，视觉评审阶段不能显示为处理中。
  运行流必须说明当前修复次数，例如“编译修复 1/2”。生成、评审、再次迭代、重新渲染、最终导出、
  接受修订、拒绝修订、项目导入、项目导出、新建模型、本地模型导航，以及目标置信度和自动迭代控件，
  在本次修复完成或以诊断停止前都保持禁用或只读。
- 中间 Agent 运行流持续承载用户请求记录、流式生成代码、折叠后的完整代码、编译输出、渲染开始、
  渲染进度、渲染完成、渲染错误、评审总结、问题列表、置信度、修正提示词、回退事件和迭代事件。
  每条评审记录只渲染一个修正提示词区块；同一修正提示词不能在评审记录内再嵌套显示成第二张提示词卡片。
  完成后的代码默认折叠，避免占据中间面板空间；高级 OpenSCAD 编辑器仍保留在输入区下方。
- 有上限的置信度运行期间，中间 Agent 运行流展示自动评审/修订进度、最新置信度结论、
  已消耗自动迭代次数，以及停止原因：达到目标置信度、达到自动迭代上限、编译失败、评审失败、
  provider 失败或其他用户可见错误。
- 任意有上限的置信度运行停止后，应用退出禁用的自动运行状态，展示停止原因，保留最新代码、STL、
  十四张视图、评审结论、置信度和修正提示词，并说明不会再发生后续自动 LLM 调用。正常手动操作按最新项目状态恢复：
  评审仍然有效时可以最终导出，需要继续修改时可以再次迭代或编辑提示词，遇到错误后可以编辑代码或重新渲染。
- 新任务或空任务状态下，中间运行流仍然把“Agent 运行”标题文案放在面板最上方，并使用与已有对话运行流一致的容器、
  基础间距和结构，但不显示单独的“AI 思考”占位卡。首次运行时应在同一界面结构里追加记录，避免出现明显的布局跳变。
- 对话运行记录包括用户请求、AI 生成/修订、渲染工具开始/进度/完成、评审开始/结果、修正提示词就绪、
  置信度、回退事件、迭代开始和错误记录。渲染记录内部可以由 MCP/渲染适配器驱动，但用户界面文案应使用“渲染”或
  “渲染工具”，不暴露未解释的内部术语。
- 右侧结果面板只包含可交互 3D 预览、十四个具名视图和资产下载控件，不包含编译日志、
  评审文本、提示词记录或迭代历史。成功渲染后，可交互预览位于面板顶部，并且是最大的视觉区域。
  它在 Three.js canvas 中渲染当前 STL，支持指针拖动旋转、滚轮缩放和平移手势，并且保持在结果面板内，
  不让页面纵向增长。预览区域或 canvas 在英文界面使用可访问名称 `Interactive STL preview`，
  在中文界面使用对应本地化名称；空状态和无效 STL 状态也必须保留标签，键盘焦点不能被困在预览控件内。
  没有 STL 前应显示空的预览框；模型变化或组件卸载时必须释放 WebGL 资源；
  遇到无效 STL 时应干净恢复，不破坏工作台其余部分。十四个固定视图位于可交互预览下方，
  并保持评审证据顺序：正视图、后视图、左视图、右视图、俯视图、仰视图、前右上等轴测、
  前左上等轴测、后右上等轴测、后左上等轴测、前右下等轴测、前左下等轴测、后右下等轴测、
  后左下等轴测。桌面端十四个固定视图 tile 使用相同视觉尺寸，形成密集、可滚动网格；
  正视图不能放大，也不能跨越额外网格列。窄屏堆叠布局中十四个视图保持同样阅读顺序、
  相同 tile 处理和稳定可访问图片名称。在 1440 px 桌面视口下，可交互预览加固定视图网格应占据
  右侧面板可见高度的大部分，其中可交互预览是唯一被放大的模型视图。
  右侧结果面板或视图网格可以内部滚动，但整体工作台不能纵向增长到隐藏输入区、工作流操作、
  高级代码编辑入口或资产下载控件。PNG、SCAD 和 STL 下载控件必须仍能在结果面板内触达，
  且不能把日志、评审文本或提示词记录混入右侧面板。
- 每个视图 tile 使用对应角度的可见标签和图片 alt 文本：正视图、后视图、左视图、右视图、俯视图、
  仰视图、前右上等轴测、前左上等轴测、后右上等轴测、后左上等轴测、前右下等轴测、前左下等轴测、
  后右下等轴测、后左下等轴测。
  每个 PNG 下载按钮也使用匹配的可访问名称和文件名：
  `ai-openscad-front.png`、`ai-openscad-back.png`、`ai-openscad-left.png`、
  `ai-openscad-right.png`、`ai-openscad-top.png`、`ai-openscad-bottom.png`、
  `ai-openscad-iso-front-right-top.png`、`ai-openscad-iso-front-left-top.png`、
  `ai-openscad-iso-back-right-top.png`、`ai-openscad-iso-back-left-top.png`、
  `ai-openscad-iso-front-right-bottom.png`、`ai-openscad-iso-front-left-bottom.png`、
  `ai-openscad-iso-back-right-bottom.png` 和 `ai-openscad-iso-back-left-bottom.png`。
  较长的视图标签和下载按钮文案在桌面和窄屏布局中都必须不重叠、不产生横向溢出；视觉上可以换行或截断，
  但可访问名称必须保持完整。
- 只要 `currentCode` 非空，资产下载控件就包含当前 OpenSCAD 源文件下载。源文件下载的可访问名称为
  `Source SCAD`，文件名为 `ai-openscad-source.scad`。它下载当前可编辑源码，而不是最终导出时标准化后的
  高精度源码。已有的“最终导出”操作仍在用户确认后下载高精度 `.scad` 和 `.stl` 文件。
- 只要存在 OpenSCAD 代码，用户就可以通过次要的重新渲染操作编译手动编辑后的代码。
  重新渲染失败显示在中间运行流里，并保持工作台可继续操作。渲染错误必须包含可读的
  OpenSCAD 诊断信息，包括 include 文件缺失、未知/被忽略模块以及 stderr warning/error；
  UI 不能只显示数字形式的 worker 或运行时错误码。
- 渲染超时时应说明草稿超过浏览器渲染复杂度预算，模型渲染阶段保持错误状态，
  当前可编辑 OpenSCAD 不丢失，并且用户简化草稿或编辑代码后仍可重新渲染。
- 代表性的纹理模型（例如 20cm 高的波浪杯）在草稿评审时应生成低复杂度、可检查的波浪几何，
  不应使用 100 层 `linear_extrude()` 叠加、密集多边形波浪环或逐层布尔挖空。
  生成的草稿应保持在 45 秒浏览器渲染超时内，并产生全部十四张可见正交和等轴测视图。
- 同一浏览器渲染复杂度预算适用于首次草稿、基于评审的修订，以及用户编辑修正提示词后的再次迭代。
- 阶段条和操作状态同时使用文字标签与视觉样式，不只依赖颜色；键盘焦点顺序保持为左侧面板、
  Agent 面板、结果面板。对话记录通过标签和布局区分用户、AI、渲染器/工具和评审角色，而不是只依赖颜色。
- 流式输出和阶段状态更新应使用 polite live-region 或等效的可访问状态文本。折叠代码控件需要可访问名称，
  并暴露当前展开/折叠状态。
- 自动化 E2E 覆盖应通过结构、可访问性、几何和状态断言验证箭头流水线阶段条、再次迭代后的当前轮次状态重置、Codex 式对话记录、
  完成代码默认折叠且可展开、渲染开始/完成提示、常规界面不显示提示词记录、评审到修正提示词的交接、
  迭代请求包含原始需求、最新已接受代码和可编辑意见、桌面端和窄屏堆叠布局中的左侧排序、隐藏 token/重复就绪 UI、
  右侧职责、可交互预览作为唯一大图、十四个固定视图等大，以及生成、渲染、评审、待确认修订、重新渲染失败、
  再次迭代状态下的主按钮切换，包含非纯颜色状态标签和键盘焦点顺序。E2E 覆盖还应验证可交互 STL 预览是
  结果面板顶部最大的视觉区域、暴露可访问名称、可以拖动且不会改变十四张固定评审图片或视觉 provider payload，
  并且桌面和窄屏布局中 PNG/SCAD/STL 下载都能触达且拥有预期可访问名称。Playwright 像素截图检查只在本地
  视觉回归检查中执行，CI 必须跳过。
- 自动化 E2E 覆盖必须包含新的目标置信度和自动迭代控件，验证它们有稳定可访问名称、可见标签、
  可读的当前值、键盘可操作、默认值/夹取行为正确，并且桌面和窄屏堆叠布局都没有横向溢出。
  本地截图覆盖必须包含左侧设置区中可见的这些控件。
- 本地视觉回归截图必须明确覆盖右侧结果面板：渲染后可交互预览是顶部最大的视觉区域，
  十四个固定视图 tile 在稳定顺序中使用相同视觉尺寸，任何内部滚动都限制在结果面板或视图网格内，
  标签不重叠，没有横向溢出，并且 PNG/SCAD/STL 下载控件保持可见或可触达。
- 测试必须验证编译失败和危险渲染诊断只会触发有上限的编译修复文本请求，而干净渲染完成、
  视觉评审完成和图片证据在显式用户启动的有上限置信度运行之外，不会触发额外 `/api/llm` 请求。
  测试还必须验证有上限的置信度运行会在达到目标置信度或耗尽自动迭代次数时停止，并验证修订提示词不包含评审置信度。
- 测试必须验证“完成渲染但包含危险 OpenSCAD 诊断”的情况会像渲染失败一样处理：阻断视觉评审、
  在有代码模型 key 时进入有上限编译修复路径，在产生干净渲染前不把渲染阶段标为完成，
  让评审阶段保持未激活/阻塞，修复期间禁用会改变状态的控件，修复停止后恢复控件，
  持续显示并播报可读诊断，阻断 STL/最终导出下载，并且不会继续落到评审置信度缺失失败。
  最终导出测试也必须覆盖最终精度渲染返回 STL 但带危险诊断的情况，并验证不会下载最终文件。
- 自动化 E2E 覆盖必须验证评审 UI 会在评审结论中展示置信度百分比，并且每条评审记录只渲染一个
  修正提示词区块。它还必须覆盖低置信度自动后续修订退步：应用恢复 checkpoint 的 `currentCode`、
  STL、十四张固定视图、渲染证据、评审/置信度和可编辑修正意见；失败的后续修订消耗一次自动迭代，
  回退/重新评审不会返还该次数；任何后续自动修订前，会为恢复后的 checkpoint 发送一次新的 `/api/vision`
  评审；后续 `/api/llm` 请求不能使用较低置信度草稿或它的修正提示词；修订提示词仍然省略置信度、
  目标和阈值指令。
- 本地 E2E 渲染覆盖必须包含用户反馈的完整 20cm 波浪杯 OpenSCAD 文件，包括逐层
  `linear_extrude()` 波浪环，验证浏览器渲染能完成，并确认多角度渲染视图包含可见模型像素。
  测试必须断言十四个具名视图按顺序出现、每张图片都有非背景模型像素，并且每个 PNG 下载控件有预期可访问名称。

## 功能清单

### 生成

- 使用代码导向系统提示词，要求输出合法、确定、完整的 OpenSCAD。
- 将生成的 OpenSCAD 视为必须先编译验证的完整 artifact。编译诊断可以触发有上限的
  自动编译修复生成；渲染视图会成为评审与后续用户确认迭代，或用户启用的有上限置信度迭代的证据，
  但不能触发无上限的基于图片自动重写循环。
- 将实用的 `lich-3D/SCAD` 可打印建模 skill 内置到文本 LLM 提示词中。网页应用
  运行时不会读取本机 Codex skill 目录；产品提示词会直接包含从源码提炼出的建模
  模式、可打印性规则、公差、BOSL2 偏好和实物零件工作流。
- 以毫米为建模单位。
- 要求命名参数、稳定 CSG、清晰模块边界和可打印几何。
- 草稿生成提示词包含浏览器渲染复杂度预算：常规迭代生成的代码应避免大量薄层
  `linear_extrude()` 叠加、密集装饰阵列、逐层布尔运算和过高分段数。纹理或波浪表面
  在草稿评审时应使用低段数、可检查的近似形体。最终导出会以最终精度重新编译已接受源码；
  它不会自动重新生成更昂贵的模型。
- 首次生成和基于评审的修订文本请求都会使用同一套可打印建模 skill 上下文，
  让迭代保留可打印壁厚、间隙、打印方向和装配约束，而不只是匹配视觉评审。
- 检测中文或英文输入，并要求模型使用同一自然语言输出反馈和注释。
- 将 OpenAI 兼容的流式响应片段同时写入对话运行流和可编辑代码状态。

### 渲染

- 在浏览器中使用 `openscad-wasm`。
- 可用时在 Web Worker 中执行编译任务。
- 浏览器 STL 生成优先使用 OpenSCAD Manifold 后端；如果 Manifold 失败，则安全回退到默认后端。
- 保持一个持久化渲染 Worker，并在多次渲染之间复用已初始化的 OpenSCAD WASM 实例。
  工作台会预热该 Worker，减少首次可见渲染需要承担的初始化成本。
- OpenSCAD WASM 运行时按单次编译使用：Worker 保持持久化，但每个已初始化的
  OpenSCAD 实例只消费一次编译，并尽力清理后释放引用，随后切换为后台预热的下一个实例。
  这样避免同一个 WASM runtime 首次 `callMain()` 成功、后续重新渲染抛出非文本运行时错误码的不稳定性。
- 默认渲染超时时间为 45 秒。
- 常规迭代使用草稿精度，将 `$fn` 标准化为 32。
- 从生成的 STL 捕获十四张多角度视图。相机方向是从模型中心指向相机的稳定向量；
  正/后/左/右视图使用屏幕向上 `+Z`，俯/仰视图使用屏幕向上 `+Y`，等轴测视图默认使用屏幕向上 `+Z`，
  除非该方向与相机方向平行：
  - 正视图
  - 后视图
  - 左视图
  - 右视图
  - 俯视图
  - 仰视图
  - 前右上等轴测
  - 前左上等轴测
  - 后右上等轴测
  - 后左上等轴测
  - 前右下等轴测
  - 前左下等轴测
  - 后右下等轴测
  - 后左下等轴测
- 使用 Three.js 光照和 STL 解析生成 PNG data URL。
- 右侧结果面板使用同一份当前 STL 渲染本地 Three.js 可交互预览。该预览只是用户可拖动检查的 UI，
  不会加入视觉评审 provider payload。
- 稳定视角方向：
  - `front`：相机方向 `(0, -1, 0)`，up `(0, 0, 1)`
  - `back`：相机方向 `(0, 1, 0)`，up `(0, 0, 1)`
  - `left`：相机方向 `(-1, 0, 0)`，up `(0, 0, 1)`
  - `right`：相机方向 `(1, 0, 0)`，up `(0, 0, 1)`
  - `top`：相机方向 `(0, 0, 1)`，up `(0, 1, 0)`
  - `bottom`：相机方向 `(0, 0, -1)`，up `(0, 1, 0)`
  - `isoFrontRightTop`：相机方向 `normalize(1, -1, 0.75)`
  - `isoFrontLeftTop`：相机方向 `normalize(-1, -1, 0.75)`
  - `isoBackRightTop`：相机方向 `normalize(1, 1, 0.75)`
  - `isoBackLeftTop`：相机方向 `normalize(-1, 1, 0.75)`
  - `isoFrontRightBottom`：相机方向 `normalize(1, -1, -0.75)`
  - `isoFrontLeftBottom`：相机方向 `normalize(-1, -1, -0.75)`
  - `isoBackRightBottom`：相机方向 `normalize(1, 1, -0.75)`
  - `isoBackLeftBottom`：相机方向 `normalize(-1, 1, -0.75)`
- 草稿捕获遵守视觉评审的浏览器 payload 预算。每张 PNG 都限制在配置的评审捕获尺寸内并编码为 data URL；
  如果合并后的 provider 请求会超过浏览器/网关 payload 预算，应用会在调用视觉端点前停止，
  在中间运行流显示可读错误，并保留重新渲染/重试能力。

### 评审和迭代

- 将原始需求、当前 OpenSCAD 和多角度渲染图片发送给视觉端点。
- 只有十四个视图 key 都有非空图片时才允许视觉评审。发送给 provider 的图片数组必须遵循稳定顺序：
  front、back、left、right、top、bottom、isoFrontRightTop、isoFrontLeftTop、isoBackRightTop、
  isoBackLeftTop、isoFrontRightBottom、isoFrontLeftBottom、isoBackRightBottom、isoBackLeftBottom。
  如果捕获失败或少于十四张视图可用，应用停留在渲染/错误状态，
  不调用视觉端点。
- 旧三视图或六视图项目、13/14 这类部分捕获结果，以及绑定到不完整当前视图的旧评审，
  都必须阻断评审、最终导出和再次迭代，直到当前代码重新渲染出全部十四张视图。
- 评审请求包含最新编译/渲染证据，让视觉模型同时检查视觉匹配和明显的 artifact
  质量问题。该反馈生成用户可编辑的修正提示词和可展示的置信度结论；
  除非处于用户已启动的有上限置信度运行中，否则不能直接调用文本模型。
- 渲染证据是有边界的 provider contract，不是本地化 UI 文案。它包含编译状态、
  可读诊断、渲染精度、后端和已渲染视图数量。`viewCount` 等于稳定视图集合中非空图片数量，
  视觉评审时必须为 `14`。除非当前模型请求明确需要，否则不会发送 STL 内容、评审图片之外的截图或提示词记录。
- `compileStatus: success` 的渲染只有在十四张视图都存在，且 diagnostics 不包含危险 OpenSCAD
  warning 时才可用于评审。危险 warning 包括未定义运算、忽略未知变量/模块、参数转换失败、
  include/use 文件缺失、NaN 或非有限几何。即使已经产出 STL 文本和 PNG 截图，危险诊断也会把该渲染降级为可修复失败。
- 期望返回 JSON：
  - `summary`
  - `issues`
  - `correctionPrompt`
  - `confidence`
- `confidence` 会标准化为 0-1 分数保存，并在评审记录和停止结论中以百分比展示给用户。
  它只属于评审结论和停止决策状态。它不能追加进可编辑修正提示词、面向用户展示的提示词记录，
  或任何文本 LLM 生成/修订提示词。
- 如果 `confidence` 缺失、不是数字，或无法标准化到 0-1 范围，在有上限置信度运行中会被视为评审失败。
  本次运行以可读原因停止，并且不能基于该评审发送后续文本 LLM 修订。
- 当模型返回格式错误或非 JSON 评审文本时，会进行兜底处理。
- `correctionPrompt` 需要具体到文本 LLM 可以执行：它应引用原始需求、可见视图或受影响模型区域、
  观察到的不匹配、必须保留的约束、可推断的 OpenSCAD 模块或几何关系，以及可用的尺寸、位置或比例指导。
  它不能直接返回修订后的代码。
- 迭代请求会把最新已接受/已渲染的 OpenSCAD、原始需求、评审总结、问题列表和可编辑评审意见一起发送给
  文本 LLM，让下一版草稿是针对性修改，而不是无关的重新生成。修订请求必须有意省略评审置信度、
  目标置信度和停止阈值指令。
- 有上限的置信度运行始终由用户动作启动。自动迭代次数为零时，**生成** 和 **再次迭代**
  保持当前手动行为：渲染后停止，等待用户评审或迭代。自动迭代次数大于零时，**生成**
  会渲染初始草稿并自动评审；**再次迭代** 会渲染用户点击触发的修订并自动评审。
  如果最新评审置信度低于目标，每次后续自动修订消耗一次自动迭代次数。本次运行会在达到目标置信度、
  达到配置的自动迭代上限，或遇到任意编译/评审/provider 错误时停止。
- 有上限置信度运行中的后续修订会自动作为当前草稿应用，然后编译、渲染并评审。它们不会停留为需要
  “接受修订”或“拒绝修订”的待确认修订。非自动修订流程仍保留手动接受/拒绝行为。
- 如果后续自动修订成功渲染，但视觉评审置信度低于上一份 checkpoint 的置信度，该低分草稿视为退步。
  应用恢复 checkpoint，并在 Agent 运行流中记录回退事件。回退事件必须说明已恢复上一份 checkpoint，
  展示 checkpoint 置信度和退步后的较低置信度，并说明在任何后续自动修订前，正在重新评审恢复后的模型。
  使用任何后续修正提示词前，应用必须重新评审恢复后的 checkpoint。若仍有自动迭代预算，
  后续自动修订使用这份新评审意见和恢复后的 checkpoint 代码。应用不能基于置信度降低的草稿或它的修正提示词继续迭代。
- 如果初始草稿、用户点击触发的再次迭代，或自动后续修订在有上限置信度运行中编译失败，
  或产生危险渲染诊断，在有代码模型 key 时，该编译步骤必须在视觉评审前使用已有的有上限编译修复循环。
  编译修复文本 LLM 调用独立于评审驱动的自动迭代次数，不消耗自动迭代次数。
  评审驱动的自动修订请求开始时会消耗一次自动迭代次数，即使它后续编译时需要修复。
  如果缺少代码模型 key、编译修复次数耗尽，或修复后的代码仍无法得到干净渲染，
  有上限置信度运行以渲染诊断停止，不进入视觉评审，也不再发起另一轮评审驱动修订。
- 每个进行中的有上限置信度运行都绑定到当前项目 id、当前代码版本和该运行生成的十四视图渲染证据。
  浏览器刷新、导入、切换项目、新建模型、手动编辑代码、手动重新渲染、接受/拒绝待确认修订，或任何项目状态重置都会取消本次运行。
  已取消运行中迟到的生成、修订、编译修复、渲染或评审响应必须被忽略，不能更新项目，不能启动另一轮渲染/评审/provider 请求，
  也不能覆盖可见运行结论。
- 生成或手动编辑代码导致的编译失败和危险渲染诊断会继续显示在运行流中，并在有代码模型 key 时
  自动折叠进有上限的编译修复文本请求。达到修复上限后，同一份诊断指导仍保留在可编辑输入区供用户手动重试。视觉评审
  发现仍必须等用户选择再次迭代操作后，文本模型才会运行；唯一例外是用户已经启动的有上限置信度运行中的后续修订。
- 本地项目数据中仍保存生成、编译、评审、修订和最终导出的提示词记录，用于导出/调试；
  但普通用户在工作台常规界面中看不到提示词记录面板。

## 系统架构

- 前端：React、Vite 和 TypeScript。
- 渲染：`openscad-wasm` 在浏览器中编译 OpenSCAD，优先放在 Web Worker 中执行。
  Three.js 解析 STL 输出、捕获多角度 PNG 视图，并在结果面板渲染可交互预览。
- 模型网关：Cloudflare Pages Functions，位于 `functions/api`。
- 持久化：浏览器 `localStorage`；没有服务端项目数据库。
- 部署：Cloudflare Pages 项目 `ai-openscad`。

重要源码位置：

- `src/App.tsx`：主工作台 UI 和用户工作流状态。
- `src/lib/apiClient.ts`：LLM、视觉评审、修订和 token 估算客户端。
- `src/lib/models.ts`：支持的模型预设和 provider 路由。
- `src/lib/render.ts`：浏览器 OpenSCAD 编译和渲染适配器。
- `src/lib/renderWorker.ts`：worker 编译路径。
- `src/lib/capture.ts`：STL 到多角度 PNG 捕获。
- `src/lib/project.ts`：项目持久化、导入和导出。
- `src/lib/i18n.ts`：英文和中文 UI 文案。
- `functions/_shared/modelGateway.ts`：MiMo 和 DeepSeek provider 代理。
- `tests/`：Playwright 工作流覆盖，包含结构、状态、几何和可访问性断言。

## 数据模型

核心项目状态包含：

- 项目身份：`id`、`title`、`updatedAt`
- 用户输入：`requirement`、`originalRequirement`
- 模型设置：`codeModelId`、`visionModelId`
- 模型资产：`currentCode`、`proposedCode`、`stl`、`views`
- 运行输出：`compilerOutput`、`renderEvidence`、`review`、`runEvents`
- 审计轨迹：`iterations`、`promptTrace`

浏览器 `localStorage` 中还保存工作台偏好状态：

- `targetConfidencePercent`：整数百分比，默认 `85`，夹取到 `1-100`
- `autoIterationLimit`：整数，默认 `0`，夹取到 `0-5`

这些偏好是本地 UI 预设，不是项目 artifact。项目 JSON 导入/导出包含评审置信度和运行事件，
但不会导出或恢复进行中的有上限置信度运行。active run 状态只保存在内存中；页面刷新、项目导入、
项目切换或新建模型都会取消它，恢复后绝不会继续自动调用 provider。

`views` 是按稳定 `ViewKey` 顺序组织的对象：

- `front`
- `back`
- `left`
- `right`
- `top`
- `bottom`
- `isoFrontRightTop`
- `isoFrontLeftTop`
- `isoBackRightTop`
- `isoBackLeftTop`
- `isoFrontRightBottom`
- `isoFrontLeftBottom`
- `isoBackRightBottom`
- `isoBackLeftBottom`

每个值都是 PNG data URL 或空字符串。UI、下载、渲染证据和视觉 payload 都使用同一顺序。
从旧三视图或六视图项目导入时，缺失 key 会填为空字符串，直到用户重新渲染。

除非用户显式导出项目 JSON，或通过网关调用外部模型 provider，否则数据都保留
在浏览器本地。

## API 和 Provider 行为

前端调用：

- `POST /api/llm` 用于代码生成和修订。
- `POST /api/vision` 用于视觉评审。

两个端点都是 Cloudflare Pages Functions，会把 OpenAI 兼容的 chat completion
请求代理到配置的 provider。

支持的 provider：

- MiMo：`https://api.xiaomimimo.com/v1`
- DeepSeek：`https://api.deepseek.com`

认证行为：

- 如果浏览器提供 API Key，网关会将其转发给 provider。
- MiMo 可以回退使用 Pages 环境中的 `MiMo_KEY` 或 `MIMO_KEY`。
- DeepSeek 在当前应用中没有托管回退 Key。

当前模型预设：

- 代码模型：
  - MiMo V2.5 (`mimo-v2.5-pro`)
  - DeepSeek V4 Pro (`deepseek-v4-pro`)
- 视觉模型：
  - MiMo V2.5 (`mimo-v2.5`)

浏览器中填写的 API Key 会保存在 `localStorage`，并作为 bearer token 发送给
Pages Function 网关。

## 质量标准

应用应保持这些保证：

- 桌面端主工作流不应隐藏关键操作。
- 草稿生成和评审过程要展示进度。
- 无效 OpenSCAD 会显示错误，并保持页面可继续使用。
- 只有完成干净成功渲染、具备所需视图且没有危险诊断后才可下载 STL；只要有当前代码即可下载源 SCAD。
- 评审请求必须包含渲染图片。
- 评审不会触发文本生成，除非它属于用户明确启动且仍有自动迭代次数的有上限置信度运行。
- 新迭代会清除过期评审状态。
- 文本 LLM 请求包含应用内置的可打印建模 skill 上下文，因此提示词记录可以解释
  生成代码为什么偏向实用 OpenSCAD 模块、显式公差和 3D 可打印几何。
- CI E2E 测试通过结构和几何断言保护桌面工作台布局，以及窄屏堆叠布局中的左侧排序；Playwright 截图断言只在本地执行。

## 发布检查

生产部署前：

```bash
npm test
npm run test:e2e
npm run build
```

部署目标：

- Cloudflare Pages 项目：`ai-openscad`
- 生产域名：`https://ai.openscad.tech`
- GitHub Actions workflow：`.github/workflows/deploy.yml`
  - Pull request 会运行 `npm test`、`npm run test:e2e` 和 `npm run build`。
    clean runner 会使用 `npm ci` 安装依赖、设置 Node 24，并通过
    `npx playwright install --with-deps chromium` 安装 Chromium。
    `node-version` 设置控制项目命令使用的 Node；每个 `uses:` action 也应保持在
    其 action metadata 运行于 Node 24 runtime 的版本，避免废弃的 Node 20
    runtime 警告。
  - 推送到 `main` 时会执行同样检查，然后使用
    `npx wrangler pages deploy dist --project-name ai-openscad --branch main` 部署。
  - 手动 `workflow_dispatch` 生产部署只允许 `main` 分支，并使用同一部署命令；
    从其他 ref 手动运行时只执行检查并跳过生产部署。
  - Cloudflare Pages 项目 `ai-openscad` 的生产分支必须是 `main`。
  - `deploy / checks` job 应配置为合并到 `main` 前的必需 branch-protection 状态检查。
  - 必需的 GitHub 仓库 Secrets：
    - `CLOUDFLARE_API_TOKEN`
    - `CLOUDFLARE_ACCOUNT_ID`
  - `CLOUDFLARE_API_TOKEN` 必须具备编辑/部署目标 Cloudflare Pages 项目的权限，
    账号由 `CLOUDFLARE_ACCOUNT_ID` 指定。
- 可选 Cloudflare Pages 环境变量：`MiMo_KEY` 或 `MIMO_KEY` 仅在用户没有提供
  浏览器 Key 时使用。它在 Cloudflare Pages 中配置，不是 GitHub Actions secret。

部署后：

- 确认生产页面返回 HTTP 200。
- 在 provider 凭证可用时，对生成、草稿渲染、评审和导出流程做 smoke test。
